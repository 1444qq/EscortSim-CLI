# EscortSim 设计历史与思路

> 本文档记录设计讨论的演变过程，供后续开发参考。

---

## 2024-05-14 第一次设计讨论

### 起点：核心乐趣定义

来自用户的脑图：

```
核心乐趣：发展、培养、规划、谈判、斡旋

背景：镖局管理器
  - 招募和培养镖师（性格迥异、能力各异、角色成长）
  - 发展镖局（开辟镖路、拓展业务、拓展势力范围）
  - 维护地区关系（官府、地方势力、绿林好汉、土匪、武林门派）

亮点预估：
  1. 镖师的真人感（AI+AIGC驱动，武侠小说知识库）
  2. 规划和谈判的真实感（自然语言设定策略，镖师自主执行）

玩法循环：
  招募 → 扩大范围 → 面对挑战 → 承受损失 → 投入/更改策略 → 成长 → 扩大
  （类公司运营+宝可梦收集培养+AI驱动类辩论赛体验）
```

### 设计决策 #1：技术选型

**决定**：不用 UE5，用最轻量的方式快速验证。选定 Node.js CLI + 本地 Ollama (qwen3-coder:30b)。

**理由**：核心乐趣是"交涉策略"，不需要图形引擎来验证。

### 设计决策 #2：先验证什么

**决定**：先验证"AI 交涉是否有趣"。

**理由**：
- 经营循环 → 已被无数游戏验证，一定可行
- 角色养成 → 宝可梦/RimWorld已验证，一定可行
- AI驱动的谈判斡旋 → 从未被验证，这是最大赌注和最大亮点

### 设计决策 #3：玩家定位

**决定**：90% 幕后指挥 + 关键时刻可介入。

**后续修正**（同日）：去掉回合内介入。玩家只在出发前设定策略，交涉全自动。

**理由**：如果玩家可以随时介入，策略设定就没意义了。反馈循环是：
```
设定策略 → 看结果 → 发现问题 → 调整策略 → 下次更好
```
介入打断了这个循环。

### 设计决策 #4：约束产生意义

**发现**：没有约束的决策没有重量。"能花钱解决就花钱"在有1000两和只剩80两时含义完全不同。

**加入三层约束**：
1. 世界状态（大宋宣和二年，兵力空虚，最后旺季）
2. 镖局状态（180两家底，100两欠债，月开支45两）
3. 任务约束（报酬120两 vs 违约赔偿80两，时间限制）

### 设计决策 #5：不让玩家介入交涉过程

**决定**：交涉全自动推演 + 事后策略复盘。

**理由**：控制玩家输入范围更有利于建立反馈循环。玩家的学习点在"我的策略设定是否合理"，而不是"我能不能亲自把话说好"。

### 设计决策 #6：日常事件 = 无损测试（重大转折）

**问题**：当前原型一上来就是高压场景，玩家没有安全空间去认识自己的人。

**洞察**：
- 玩家需要"单元测试"——低风险日常事件来观察镖师行为
- "处事风格"本质是玩家打磨的提示词
- 日常事件暴露问题 → 玩家调整规矩 → 看效果 → 准备好了再出镖

**参考**：
- RimWorld：通过日常观察理解小人性格，大事件来时才知道谁靠得住
- 足球经理：赛前设定战术 → 看上半场效果 → 中场调整 → 看下半场

**新玩法循环**：
```
日常经营（低风险，建立认知）
  → 观察镖师行为
  → 调整"处事风格"（给镖师/镖局下规矩）
  → 验证调整效果
  → 觉得准备好了 → 出镖

出镖（高风险，验证认知）
  → 多段路程 + 中间调整窗口
  → 梯度结果（不是成功/失败二元论）

复盘 → 回到日常继续打磨
```

### 设计决策 #7：策略输入结构化

**从**：纯自然语言（"能花钱解决就花钱"）

**改为**：结构化优先级 + 自然语言补充
```
遭遇应对优先序：谈判 > 花钱 > 搬关系 > 威慑 > 绕路 > 武力
花钱上限: 50 两
时间态度: 赶时间
人员安全: 最高优先
补充: ___（自然语言）
```

**加上针对个人的"规矩"**：
```
【莽夫李的规矩】
  · 铁嘴张说话的时候你闭嘴
  · 除非对方先动手否则别动手
```

### 设计决策 #8：任务结果是频谱

**从**：成功/失败

**改为**：梯度频谱
```
圆满 → 顺利 → 磕碰 → 勉强 → 部分失败 → 出事 → 灾难
```

每次玩家的心态不是"能不能过"，而是"这次能做到什么程度"——总有优化空间。

## 核心设计原则（总结）

1. **约束产生意义** —— 资源有限、时间有限、人有脾气
2. **间接控制** —— 你设定规矩，看别人执行，在偏差中学习
3. **日常即练习** —— 低风险事件是"单元测试"，高风险出镖是"上线"
4. **性格即变量** —— 镖师性格让策略执行产生偏差，这是核心乐趣
5. **梯度结果** —— 没有完美，只有更好。总有优化空间
6. **过程可读** —— 玩家能看懂"为什么这次是这个结果"，建立因果认知

---

### 设计决策 #9：限制系统（v3）

**加入三种限制**：
1. 规矩槽位有限（总规矩3条，个人2条，从预设选项选，选A不能选B）
2. 时间压力（每天扣3两，30天入冬，不出镖坐吃山空）
3. 镖师会抗拒（性格冲突大的规矩，服从度低，可能不执行）

**没选的**：适应期（改了规矩不立刻生效）—— 等待感无聊。

### 设计决策 #10：发现问题——太像 DND 了

**问题**：颗粒度太细。管一个镖师在一次对话中说什么 = DND/跑团的尺度。

**但用户想要的是**：公司运营/城市建设类心流。尺度应该是经营一个组织的方向，不是指导每一场对话。

**滑向 DND 的原因**：
- "莽夫李说话时你闭嘴" = 战术级，管一个人一次行为
- 应该是 "这个月重点搞好官府关系" = 战略级，管镖局方向

### 设计决策 #11：纯文本 CLI 不够

**问题**：
- 状态一览性差——看不到全局
- 操作感缺失——1234菜单没有"管理"的感觉
- 反馈不直觉——"声望+5"不如看到一个条在涨
- 人物缺乏存在感——没面孔

**结论**：需要视觉操作和反馈。但不需要 UE5 级别。

**待定**：具体做到什么程度（极简卡片 / 信息面板 / 有地图有实体）

## 2026-05-27 设计决策 #12：Token 即游戏资源

### 起因

CLI 原型已迁移为 Electron 桌面应用，支持 OpenRouter BYOK。既然运行时消耗真实 Token，何不把 Token 消耗本身变成游戏机制？玩家的目标不只是"赚钱还债"，还包括"用最少的 Token 高效通关"。

### 核心理念

**1. 任务解决的 Token 经济**
- 玩家给出清晰、精准的规矩/策略 → AI 一两轮就解决 → 省 Token
- 玩家交代不清/前后矛盾/没定位问题 → AI 需要多轮澄清/走偏 → 费 Token
- 通过**可见的对话轮数**和**Token 消耗数字**让玩家感知代价
- 给出警告提示玩家"你的策略可能有盲区"

**2. 镖师招募的 Token 经济**
- 花费游戏内货币 + 声望 + Token 来招募新镖师
- Token 花得越多 → 角色设定越丰满（更多 bio、更复杂性格、更精细立绘）
- 给玩家"为角色买单"的感觉——投入越多越了解 TA

### 设计决策

**决定**：Token 消耗作为显性资源，和游戏内货币（银两）、声望并列成为第三维度资源。

**反馈循环**：
```
策略精准 → 少轮解决 → 省 Token → 玩家感到"我变聪明了"
策略模糊 → 多轮消耗 → 费 Token → 警告驱动优化 → 下次更省
招募投入 → 花 Token → 角色更丰满 → 更好用/更有感情
```

**类比**：
- 省 Token = 足球经理里战术板设置精准，上半场不用换人
- 费 Token = 战术有漏洞，教练反复喊话调整
- 招募花 Token = 花转会费买球员，大牌球员信息更详细

### 多轮对话触发机制

```
LLM 回复末尾标记:
[状态: 已解决]      → 事件结束，1轮搞定（最省）
[需要指示: 具体问题]  → 触发玩家选择:
                       A. 补充指示（消耗第2轮）
                       B. 放手不管（省 Token 但镖师按性格行事，风险自负）
```

### 镖师招募三档

| 档位 | Token消耗 | 产出 |
|------|-----------|------|
| 简略 | ~500 | 名字 + 基础属性 + 1 tag + 短 bio |
| 标准 | ~1500 | 完整属性 + 3 tags + tendency + 完整 bio + 规矩选项 |
| 精细 | ~3000 | 标准 + 详细前史 + 人际关系 + 独特事件线 + SVG 立绘 |

### 实施优先级

```
Phase 1: Token 追踪基础设施（让消耗可见）
Phase 2: 多轮对话机制（省Token=玩得好的核心循环）
Phase 4: 策略警告系统（出发前提醒盲区+预估消耗）
Phase 3: 镖师招募系统（花Token=买角色深度）
Phase 5: 存档 + Token 统计面板
```

### 技术要点

- OpenRouter API 返回 `usage.total_tokens`，需要在 `chatStream()` 中捕获
- Streaming 模式下 token 计数在最后一个 chunk 或需要额外查询 `/auth/key`
- 角色生成约束 LLM 输出 JSON schema，确保可解析入游戏状态
